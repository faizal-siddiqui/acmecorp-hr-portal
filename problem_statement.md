Expectations
This exercise is intentionally designed to assess how you think, design, and build software in an AI‑driven environment.
You are required to use AI tools to accelerate your work. We care about how you use them, the clarity of your thinking, and the quality of your engineering decisions.
We expect you to:

- Demonstrate clarity in thought and structured problem solving
- Show strong engineering fundamentals & product thinking
- Make thoughtful architectural and design decisions
- Write production‑quality code and tests
- Use AI intentionally while maintaining correctness and quality
  Please make incremental commits so we can understand how your solution evolved.
  Goal:
  Build a employee salary management software for an organization with 10,000 employees.

User Persona:
HR Manager of the orgProblem Statement:
Currently, ACME org’s HR team manages salary data for 10,000 employees across multiple countries, with everything managed via excels, which is tedious. We want the HR manager to manage the salaries data via web based software and able to answer questions about how the org pays people.
Requirements:
Write a one page requirements document before building the software, outlining the goal, scope & features, and what you are deliberately leaving out, and reasoning for it.

  Rubrics for grading
  Requirements: reasoning quality for what they deliberately chose to not build
  Usability: is the product intuitive to use, is/can the user persona use it without thinking too much about how this works?
  Functionality: Is this functionally complete based on the product requirements?
  Performance: (from UX point of view) - does this feel smooth and fast to use, where it matters?
  Architecture: Does the diagram explain the architecture well, and have the right tradeoffs been made?
  Standards & practices: does this follow the industry-standard practices for the stack?
  Test quality – assertion strength, mocks handling
  AI usage effectiveness – How AI is used to delegate? Any notes on that
  Documentation – developer experience. Is this obvious, and easy to get started with as a new developer onboarding to this repo?
  How to Succeed in This Assessment

  · Evidence of product thinking
